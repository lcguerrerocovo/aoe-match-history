# Patch Notes & Balance Patch Management

Fetch AoE2:DE patch notes from Steam, save verbatim as markdown, analyze for balance changes, extract per-civ summaries, and maintain `data/balance-patches.json`.

This config determines:
1. Which patch boundaries the stats-generator uses for win rate comparisons
2. Per-civ balance change summaries shown in UI stats tooltips

## Files

| File | Purpose |
|------|---------|
| `data/patches.json` | All known patches (auto-generated by `scripts/fetch-patches.mjs`) |
| `data/patch-notes/v{version}.md` | Verbatim patch notes, one per patch |
| `data/balance-patches.json` | Human-curated: balance-relevant patches with per-civ changes |

## Workflow

### Step 1: Inventory

1. Read `data/patches.json` for the full patch list
2. Read `data/balance-patches.json` if it exists — note which patches are already curated
3. List `data/patch-notes/` to see which notes are already saved
4. Identify gaps: patches without saved notes, patches not yet evaluated for balance relevance
5. Report inventory to user: "Found N patches, M have saved notes, K are in balance config. N new patches to process."

### Step 2: Fetch Missing Patch Notes

Run the fetch script (sandbox bypass needed — Steam API and ageofempires.com are not in the allowlist):

```bash
node scripts/fetch-patch-notes.mjs              # Last 6 months (default)
node scripts/fetch-patch-notes.mjs --months 12  # Last 12 months
node scripts/fetch-patch-notes.mjs --all        # All patches
node scripts/fetch-patch-notes.mjs --force      # Overwrite existing notes
```

The script:
1. Reads `data/patches.json` for the patch list
2. Skips patches that already have saved notes in `data/patch-notes/`
3. For major patches: fetches the full blog post from `ageofempires.com/news/` (has complete balance details)
4. Falls back to Steam News API BBCode content for minor/hotfix patches
5. For hotfixes mentioned in parent patch bodies: saves a stub referencing the parent
6. Converts HTML/BBCode to markdown and saves as `data/patch-notes/v{version}.md` with frontmatter

After running, verify the output: `ls -la data/patch-notes/` and spot-check a file.

### Step 3: Analyze for Balance Changes

For each patch note not yet in `balance-patches.json`, read the markdown and determine:

**Is this patch balance-relevant?** Signs of YES:
- Civilization-specific changes (unit stats, tech costs, civ bonuses, unique units/techs)
- General unit/building stat changes affecting all civs
- Map pool rotations (changes which maps appear in ranked)
- Economy changes (gather rates, building costs, villager behavior)

**NOT balance-relevant:**
- Bug fixes, crash fixes, UI/UX changes
- Campaign/scenario editor changes
- Modding tools, spectator changes
- Matchmaking infrastructure (unless ranked mode rules change)
- Art/sound/localization updates

**Per-civ change extraction** (for balance-relevant patches):
- Read through the "Balance" or "Gameplay" sections of the patch notes
- Extract a concise list of changes per affected civ
- Each entry should be specific and short: `"Knight +1 pierce armor"`, `"Throwing Axeman range -1"`, `"Team bonus now grants +2 LoS instead of +3"`
- Use civ names exactly as they appear in `data/rl_api_mappings.json`
- Changes that affect ALL civs go in `generalChanges` (e.g., "Siege Tower rework", "Mangonel minimum range added")
- If a civ was added in this patch, note it: `"New civilization added"`

**Classify impact:**
- `"major"` — significant meta shift: multiple civ changes, unit reworks, new civs added
- `"minor"` — targeted adjustments: 1-2 civ tweaks, small stat changes

### Step 4: Present Recommendations

**HARD GATE: Do NOT write to `data/balance-patches.json` without explicit user approval.**

Present each analyzed patch clearly:

```
### v{version} — {date} — {title}
**Verdict:** Balance-relevant ({impact}) | Not balance-relevant
**Reason:** One sentence explaining why

**Civ Changes:**
- {CivName}: {change1}, {change2}
- {CivName}: {change1}

**General Changes:**
- {change description}

**Proposed JSON entry:**
{formatted JSON block}
```

Group by verdict: balance-relevant first, then not-relevant (collapsed/summarized).

If `balance-patches.json` already exists and has entries, highlight:
- New patches to add
- Whether the "current" vs "previous" patch boundary would change
- Any existing entries that might need updates (e.g., if a hotfix for the same period added more balance changes)

### Step 5: Apply Approved Changes

After user confirms:
1. Write/update `data/balance-patches.json`
2. If this is a new file, remind user that the stats-generator needs updating to use it (see "Integration" below)
3. Suggest re-running the stats-generator if patch boundaries changed

## balance-patches.json Schema

```json
{
  "description": "Human-curated balance-relevant patches for stats boundaries and per-civ tooltips.",
  "patches": [
    {
      "version": 170934,
      "date": "2026-03-12",
      "title": "Update 170934",
      "impact": "major",
      "civChanges": {
        "Romans": [
          "Knight +1 pierce armor",
          "Centurion cost reduced by 10 gold"
        ],
        "Franks": [
          "Throwing Axeman range reduced by 1"
        ]
      },
      "generalChanges": [
        "Siege Tower rework: can no longer garrison while moving",
        "Map pool: Arabia replaced with Runestones"
      ]
    }
  ]
}
```

**Rules:**
- Patches ordered by date ascending (oldest first)
- Civ names MUST match `data/rl_api_mappings.json` exactly (check spelling)
- Stats-generator uses the two most recent entries as current/previous boundaries
- `civChanges` feeds UI tooltips — keep entries concise and specific
- `generalChanges` for non-civ-specific balance changes
- `impact` is informational (not used by stats-generator currently, but useful for UI)

## Integration Points

### Stats-generator (`jobs/stats-generator/src/mappings.ts`)
Currently uses `findMajorPatches()` which filters `patches.json` by `type === 'major'`. After creating `balance-patches.json`:
- Add `loadBalancePatches()` that fetches `balance-patches.json` from GCS
- Take the two most recent entries as current/previous boundaries
- Fall back to `findMajorPatches()` if the file doesn't exist (graceful rollout)
- Deploy workflow needs `balance-patches.json` added to GCS upload

### UI StatsPage
The `civChanges` from the current patch entry feeds into per-civ tooltips:
- Tooltip on each civ row showing what changed for that civ in the current patch
- `generalChanges` could display in the page header or as a separate expandable section

## Notes

- Steam News API returns BBCode. Patch notes can be very long (5000+ words for major updates). Save the full content — the analysis extracts the balance-relevant parts.
- Some hotfixes share a date with their parent patch. Use version number as the unique identifier, not date.
- Hotfixes sometimes contain balance changes (e.g., emergency nerfs after a DLC launch). Don't skip them just because they're labeled "hotfix".
- RoR (Return of Rome) civs and changes are mixed into the same patch notes. Include them if they affect ranked matchmaking.
